# 在线调试功能（Playground）技术分析文档

> 本文档详细分析 DTS Cloud Docs 项目中"在线调试"功能的代码架构、数据流和关键实现，帮助理解和维护该功能。

---

## 一、功能概述

在线调试台是一个嵌入在文档站点中的 **浏览器端 API Playground**，允许用户：

1. **连接真实 DTS 服务器** — 通过 SDK 连接云渲染服务器（视频流）或 WebSocket API 服务
2. **实时编写和执行代码** — CodeMirror 编辑器 + Ctrl+Enter 即时执行
3. **浏览 API 示例** — 左侧树形导航，点击即填入/执行代码
4. **参数可视化调节** — 浮动面板，拖滑块实时改代码并重跑
5. **AI 代码生成** — 集成 13 个 LLM 提供商，自然语言生成 SDK 代码
6. **代码分享** — URL hash 编码代码，一键复制分享

## 四、架构设计

### 4.1 页面级路由设计

Playground 是一个 **全屏独立页面**，不使用文档站的 Header/Sidebar 布局。路由判断在 `App.jsx` 中完成：

```jsx
// src/App.jsx
const isPlayground = location.pathname === '/playground'

if (isPlayground) {
    return <PlaygroundPage /> // 全屏，无 Header/Sidebar
}

return (
    <div>
        <Header />
        <Routes>...</Routes>
    </div>
)
```

**为什么这样做？** 调试台需要 100vh 全屏高度来同时显示编辑器、控制台和视频流，文档站的 Header（64px）和 Sidebar（264px）会挤占空间。

## 五、核心模块详解

### 5.1 SDK 工具模块（sdk.js）

**文件：** `src/utils/sdk.js`

将 SDK 相关逻辑从主组件中抽离，职责单一：

| 函数                                           | 用途                                         |
| ---------------------------------------------- | -------------------------------------------- |
| `loadScript(src)`                              | 动态注入 `<script>` 标签加载 JS 文件         |
| `sleep(ms)`                                    | Promise 版延时（示例代码依赖）               |
| `escapeHtml(s)`                                | XSS 防护，SDK 回调内容转义                   |
| `matchCmdLog(text)`                            | 匹配日志中的 `__command` JSON 命令           |
| `calcTimeDiffs(arr)`                           | 计算相邻时间戳差值（日志回放用）             |
| `injectGlobalFunctions(writeLog, clearScreen)` | 注入 `window.log`、`window.sleep` 等全局函数 |
| `loadSdk(baseUrl, writeLog)`                   | 加载 ac_conf.js + ac.min.js                  |
| `destroySdk()`                                 | 销毁 SDK 实例，清理 DOM                      |
| `initConnection(opts)`                         | 根据模式初始化云渲染或 WebSocket 连接        |

**SDK 加载流程：**

```
1. injectGlobalFunctions() → 注入 window.log / sleep 等
2. loadSdk() → 加载 ac_conf.js → 加载 ac.min.js
3. initConnection() → 根据 isCloud 选择连接方式
4. apiOptions.onReady → 设置 status='ready'，触发自动执行
```

**两种连接模式：**

| 模式      | 类                  | 用途                     |
| --------- | ------------------- | ------------------------ |
| 云渲染    | `DigitalTwinPlayer` | 视频流 + API，有实时画面 |
| WebSocket | `DigitalTwinAPI`    | 仅 API，无视频流         |

### 5.2 代码执行机制

**文件：** `PlaygroundPage.jsx`

用户代码通过 `window.eval` 执行，包裹在 async IIFE 中以支持顶层 await：

```javascript
window.eval('(async ()=>{' + codeRef.current + '})()')
```

**为什么用 eval 而不是 iframe 沙箱？**

1. SDK 实例（`window.fdapi`）在主页面上下文中创建
2. 用户代码需要直接调用 `fdapi.camera.set(...)` 等方法
3. 所有必要的全局变量（`fdapi`、`fdplayer`、`log`、`sleep`）已经暴露在 `window` 上

**全局函数注入（在 sdk.js 中）：**

```javascript
window.log = (msg, noLineBreak, color) => writeLog(msg, noLineBreak, color)
window.clearScreen = clearScreen
window.sleep = sleep // ms => new Promise(r => setTimeout(r, ms))
window.exeFunction = (fn, ms) => setTimeout(fn, ms)
```

### 5.3 日志系统

**文件：** `PlaygroundPage.jsx` + `ConsolePanel.jsx`

日志系统直接操作 DOM（`innerHTML`），不是 React 状态驱动：

```javascript
const writeLog = useCallback((msg, noLineBreak, color) => {
    const el = infoPanelRef.current
    if (!el || !flagsRef.current.logEnabled) return
    // 自动清屏：超过 100 条自动清空
    if (flagsRef.current.autoClear && ++logTimesRef.current > 100) {
        logTimesRef.current = 0
        el.innerHTML = ''
    }
    const html = color ? '<font color="' + color + '">' + msg + '</font>' : String(msg)
    el.insertAdjacentHTML('beforeend', html + (noLineBreak ? '' : '\n'))
    el.scrollTop = el.scrollHeight + 100
}, [])
```

**为什么用 innerHTML 而不是 React 状态？**

1. SDK 的 `onLog` 回调可能高频触发（每帧一条日志）
2. React 状态更新会触发重渲染，高频日志会造成性能问题
3. `insertAdjacentHTML` 是原生 DOM API，不触发 React reconciliation

### 5.4 CodeMirror 智能补全

**文件：** `EditorPanel.jsx`

自定义 CodeMirror 扩展，为 `fdapi.*` 提供方法补全：

```javascript
function fdapiCompletions(context) {
    // fdapi.camera.loo → 方法补全
    let m = context.matchBefore(/(?:fdapi|api)\.([\w$]+)\.([\w$]*)$/)
    if (m) {
        const ns = m.text.replace(/^(?:fdapi|api)\./, '').split('.')[0]
        const list = API_COMPLETIONS.ns[ns] // 从 api-completions.js 获取
        return { options: list.map(o => ({ label: o.label, type: 'method' })) }
    }
    // fdapi.cam → 命名空间 + 根方法补全
    m = context.matchBefore(/(?:fdapi|api)\.([\w$]*)$/)
    // ...
}
```

**数据来源：** `api-completions.js`（131KB），由 `gen_dts.py` 脚本从 API 文档自动生成。

### 5.6 ParamPanel 参数面板

**文件：** `src/components/Playground/ParamPanel.jsx`

lil-gui 风格的浮动参数面板，核心逻辑：

1. **解析代码** — 用正则扫描编辑器代码中的 `fdapi.<ns>.<method>(...)` 调用
2. **提取参数** — 解析对象字面量 `{ color: '#ff0000', size: 10 }` 或位置参数
3. **匹配元数据** — 从 `param-meta.js` 获取参数类型、范围、枚举值
4. **渲染控件** — 根据类型渲染：滑块（number+range）、颜色选择器（color）、复选框（bool）、下拉框（enum）
5. **回写代码** — 修改控件值后，按原始偏移量把新值拼回代码字符串

```
代码: fdapi.marker.add({ color: '#ff0000', size: 10 })
                    ↓ 解析
控件: colorpicker(#ff0000) + slider(10, min=1, max=100)
                    ↓ 用户拖动滑块
回写: fdapi.marker.add({ color: '#ff0000', size: 50 })
```

### 5.7 AIChat AI 助手

**文件：** `src/components/Playground/AIChat.jsx`

集成 13 个 LLM 提供商的代码生成助手，核心特性：

- **多提供商支持：** Claude、ChatGPT、DeepSeek、通义千问、Kimi、Gemini 等
- **API 白名单验证：** 检查 AI 生成的 `fdapi.*` 调用是否真实存在，防止臆造 API
- **命名空间感知：** 根据用户问题自动匹配相关 API 签名注入 prompt
- **流式输出：** SSE（Server-Sent Events）实时显示生成过程
- **Key 本地存储：** API Key 存在浏览器 localStorage，不上传服务器

**白名单验证流程：**

```javascript
// 1. 从 api-completions.js 构建全量 API 集合
// 2. 从 api_examples.js 补充额外的 fdapi 调用
// 3. AI 生成代码后，检查所有 fdapi.xxx.yyy 调用是否在白名单内
// 4. 不在白名单的调用 → 标记为"可能存在错误"警告用户
```

## 八、关键设计决策

### 8.1 为什么全屏独立渲染？

调试台需要同时显示：导航树（264px）+ 视频流 + 编辑器 + 控制台。文档站的 Header（64px）和 Sidebar（264px）会占用约 328px 的水平/垂直空间，在小屏幕上严重影响可用性。

### 8.2 为什么用 window.eval 而不是 iframe？

1. SDK 实例（`window.fdapi`）在主页面上下文中创建
2. 用户代码需要直接调用 `fdapi.camera.set(...)` 等方法
3. 如果用 iframe，需要跨 iframe 通信，增加复杂度且难以访问 SDK 实例

### 8.3 为什么日志用 innerHTML 而不是 React？

SDK 的 `onLog` 回调可能每帧触发一次（60fps = 60条/秒）。React 状态更新会触发 reconciliation，在高频写入场景下性能不可接受。`insertAdjacentHTML` 是原生 DOM 操作，绕过 React 直接写入 DOM。

### 8.4 为什么拆分成多个组件？

重构前 PlaygroundPage.jsx 约 870 行，职责过重。拆分后：

- **主组件（360行）** 只负责状态管理和业务逻辑
- **UI 组件** 各自封装，只关心渲染，不关心状态来源
- **工具模块（sdk.js）** 纯函数，可独立测试
- 新增/修改某个区域时，只需关注对应文件

## 十二、性能优化

1. **示例数据懒加载** — `real-examples.js`（740KB）通过 `import()` 动态加载，不阻塞首屏渲染
2. **搜索索引预热** — 鼠标悬停搜索框时才加载搜索库
3. **防抖搜索** — 输入搜索关键词时 200ms 防抖
4. **日志自动清屏** — 超过 100 条自动清空，防止 DOM 节点过多
5. **代码自动保存** — 500ms 防抖写入 localStorage，防止频繁写入
6. **useRef 优化** — `codeRef`、`flagsRef` 等使用 useRef 避免不必要的重渲染

#### question01

其中aichat功能是有做什么限制吗，特别使用fdapi来写代码

1. API 白名单验证（最严格的限制）

AI 生成的代码中所有 fdapi.\* 调用都会被检查是否真实存在：

AI 生成: fdapi.camera.set(1, 2, 3)
↓ 白名单检查
fdapi.camera.set ✓ 存在 → 放行

AI 生成: fdapi.magic.doSomething()
↓ 白名单检查
fdapi.magic.doSomething ✗ 不存在 → 整条回复被拦截

白名单来源：

- api-completions.js — 文档全量 API（权威）
- api_examples.js — 补充额外的真实调用

拦截后提示：

▎ ⚠️ 生成代码中包含 SDK 文档里不存在的 API：fdapi.xxx.yyy
▎ 可能是模型臆造的接口，该响应已被拦截，请换个描述方式重新发送。

2. System Prompt 约束

prompt 中明确要求：

- "禁止臆造参数"
- "必须严格按其参数顺序、数量与类型生成"
- "不确定的参数用注释标注 TODO，切勿编造"
- 列出了所有可用的 fdapi.\* 命名空间和方法

3. 需要用户自己提供 API Key

AI 助手不自带 Key，需要用户在界面上配置：

const PROVIDERS = {
claude: { keyPlaceholder: 'sk-ant-api03-...', ... },
openai: { keyPlaceholder: 'sk-proj-...', ... },
deepseek: { keyPlaceholder: 'sk-...', ... },
qwen: { ... },
// 共 13 个提供商
}

Key 存在浏览器 localStorage（DtsAiKey\_\*），不上传服务器。

4. 命名空间感知注入

根据用户问题自动匹配相关 API 签名注入 prompt，控制 token 消耗：

const NS_ALIASES = {
camera: ['相机', '视角', '镜头', '飞行', ...],
weather: ['天气', '下雨', '下雪', '雾', ...],
marker: ['标注', '标记', '打点', ...],
// ...
}

最多注入 5 个命名空间的完整方法签名。

---

简单说：AI 不能编造不存在的 API，生成的代码会被自动校验，假接口直接拦截。 这是为了防止 AI 幻觉导致用户调用不存在的方法。

#### question02

也就是每次生成代码都需要在白名单中进行全量检索？
不是每次全量检索。白名单用的是 Set 数据结构，查找是 O(1)：

// 1. 白名单只构建一次，模块级缓存
let \_apiCachePromise = null

function fetchValidApis() {
if (!\_apiCachePromise) { // 只在第一次调用时构建
\_apiCachePromise = (async () => {
const set = buildCompletionApis() // Set 数据结构
const resp = await fetch('/api_examples.js')
// 补充额外的真实调用...
return set
})()
}
return \_apiCachePromise // 后续直接返回缓存
}

// 2. 每次 AI 生成后，只是正则提取 + Set 查找
function findInvalidApis(responseText, whitelist) {
const calls = responseText.match(/fdapi(?:\.\w+)+/g) || [] // 正则提取所有 fdapi.xxx 调用
return [...new Set(calls.filter(c => !whitelist.has(c)))] // Set.has() 是 O(1)
}

实际流程：

首次使用 → 构建 Set（~几百个 API 名称）→ 缓存
每次 AI 回复 → 正则提取 fdapi 调用（通常 3-10 个）→ 每个做 Set.has() 查找 → O(1)

所以开销很小，不会有问题。
